# Go 语言数组：设计与实现原理

> **来源**：基于《Go语言设计与实现》第三章第一节整理
> **核心观点**：Go 数组不仅是数据结构，更是**编译器优化**与**内存布局**的结合体。

## 1. 核心摘要

### 1.1 基础概念
*   **类型严格性**：数组类型由 **`[N]T`** 决定，`N`（长度）和 `T`（元素类型）缺一不可。`[3]int` 与 `[4]int` 是完全不同的类型。
*   **值语义**：数组是值类型，赋值或传参时会进行**深拷贝**（这也是切片被广泛使用的原因之一）。
*   **内存布局**：连续的内存空间，支持随机访问（O(1)）。

### 1.2 编译时优化策略
编译器根据数组长度采取不同的初始化策略，以平衡代码体积与运行效率：

| 数组长度 | 策略 | 底层实现 |
| :--- | :--- | :--- |
| **≤ 4 个元素** | **栈上直接初始化** | 拆分为独立的 `MOV` 汇编指令，无额外开销。 |
| **> 4 个元素** | **静态区拷贝** | 在静态存储区生成只读模板，运行时通过 `memmove` 拷贝到栈上。 |

### 1.3 访问与安全性
*   **编译期检查**：针对常量索引（如 `arr[1]`），编译器直接检查越界、负数等错误。
*   **运行时检查**：针对变量索引（如 `arr[i]`），编译器插入 `Bounds` 检查代码，若越界则触发 `panic: runtime error: index out of range`。

## 3. 深度解析

### 3.1 为什么长度 > 4 会触发拷贝？
根据《Go语言设计与实现》的分析，这是 Go 编译器的一种权衡（Trade-off）：
1.  **短数组**：直接展开指令效率最高，且不会显著增加代码体积。
2.  **长数组**：如果继续展开指令，会导致生成的二进制文件体积膨胀（Code Bloat）。通过在静态区存储一份模板，运行时拷贝，虽然多了一次内存拷贝的开销，但极大地减少了代码体积。

### 3.2 运行时 Panic 的代价
Go 数组的越界检查是强制的。虽然编译期能消除常量索引的检查，但对于循环中的变量索引，每次访问都会伴随一次边界判断（Branch）。这保证了内存安全，但也带来了一定的性能损耗，因此在高性能场景下，切片（Slice）配合预分配容量通常比数组更灵活。
