# Go 语言哈希表（Map）实现原理、版本演进与使用指南

![链表法](https://img.draven.co/2019-12-30-15777168478804-separate-chaing-and-get.png)

## 1. 核心数据结构

Go 的 map 基于哈希表实现，运行时主要包含三个结构体：`hmap`（表头）、`mapextra`（溢出桶管理）、`bmap`（桶）。
![hash数据结构](https://img.draven.co/2020-10-18-16030322432679/hmap-and-buckets.png)


### 1.1 hmap 结构（`runtime/map.go`）

```go
type hmap struct {
    count     int    // 元素数量，len() 返回此值
    flags     uint8  // 状态标志（如是否在写入）
    B         uint8  // 桶数量的对数，桶数 = 2^B
    noverflow uint16 // 溢出桶数量近似值
    hash0     uint32 // 哈希种子

    buckets    unsafe.Pointer // 桶数组指针，长度 2^B
    oldbuckets unsafe.Pointer // 扩容时保存旧桶
    nevacuate  uintptr        // 搬迁进度，下一个要搬迁的桶编号

    extra *mapextra // 可选字段，存储溢出桶信息
}
```

### 1.2 bmap 结构（桶）

每个桶固定存储 **8 个键值对**，超过则通过溢出桶链接。编译时根据 key/value 类型生成具体结构，概念如下：

```go
type bmap struct {
    tophash [8]uint8          // 哈希值的高8位，用于快速比较，每个key的高8位不相同
    keys    [8]keytype        // 8个键（连续存储）
    values  [8]valuetype      // 8个值（连续存储）
    overflow *bmap            // 溢出桶指针
}
```

**内存布局示意**（以 `map[int64]int8` 为例）：


```
+----------------+   +----------------+   +----------------+
| bucket[0]      |   | bucket[1]      |   | overflow bucket|
| tophash[8]     |   | tophash[8]     |   | tophash[8]     |
| keys[8]int64   |   | keys[8]int64   |   | keys[8]int64   |
| values[8]int8  |   | values[8]int8  |   | values[8]int8  |
| overflow ptr   |-->| overflow ptr   |-->| overflow ptr   |
+----------------+   +----------------+   +----------------+
```

**特点**：键和值分开存储，避免内存对齐填充浪费（例如 `map[int64]int8` 若键值交替存储会产生大量 padding）。

### 1.3 hmap 与 bmap 之间的中间管理结构：`mapextra`

`hmap.extra` 字段指向一个 `mapextra` 结构体，用于管理**溢出桶**，避免频繁内存分配并提高效率：

```go
type mapextra struct {
    overflow    *[]*bmap // 当前已使用的溢出桶地址（用于 B > 4 时的优化）
    oldoverflow *[]*bmap // 扩容时旧桶的溢出桶地址
    nextOverflow *bmap   // 下一个空闲溢出桶的地址（预分配）
}
```

**作用**：
- **预分配溢出桶**：当创建 map 或扩容时，可能会预先分配一批溢出桶（数量一般为 `2^(B-4)`），通过 `nextOverflow` 形成空闲链表，使用时直接取用，减少 `malloc` 调用。
- **快速定位溢出桶**：通过 `overflow` 切片记录所有溢出桶的指针，便于遍历和统计（如 `noverflow` 计算）。
- **扩容时的平滑迁移**：`oldoverflow` 在扩容期间保存旧溢出桶信息，待搬迁完成后释放。

这个中间层使得 Go map 在处理大量哈希冲突时性能更稳定，尤其避免了频繁内存分配带来的开销。

## 2. 哈希冲突解决

Go 使用**链地址法**解决冲突：

1. 计算哈希值，低 B 位定位桶索引。
2. 若该桶的 8 个槽位已满，则通过 `overflow` 指针创建**溢出桶**，形成链表。
3. 查找时遍历桶和溢出桶，比较 `tophash` 和完整 key。

## 3. 扩容机制
![扩容机制](https://img.draven.co/2020-10-18-16030322432579/hashmap-evacuate-destination.png)

### 3.1 触发条件

| 条件 | 说明 | 扩容类型 |
|------|------|----------|
| 负载因子 > 6.5 | `count / 2^B > 6.5`，元素过多导致查找效率下降 | **翻倍扩容**（B+1） |
| 溢出桶过多 | B ≤ 15 时溢出桶数 ≥ 2^B；B > 15 时溢出桶数 ≥ 2^15 | **等量扩容**（B 不变，整理碎片） |

### 3.2 负载因子为什么是 6.5？

- 负载因子 = 元素数量 / 桶数量。
- 太小（如 2.0）：内存浪费严重。
- 太大（如 10.0）：查找时遍历溢出桶链变长，性能下降。
- **6.5** 是 Go 团队通过 benchmark 选出的平衡点：平均每个桶存储 6.5 个元素时，溢出桶数量较少且内存利用率合理。

### 3.3 扩容策略对比

| 类型 | 新 B | 目的 | 搬迁行为 |
|------|------|------|----------|
| **翻倍扩容** | B+1 | 降低负载因子 | 旧桶键值对重新分配到新桶（原桶 或 原桶+2^B） |
| **等量扩容** | B 不变 | 整理溢出桶 | 将数据紧凑到新桶，减少溢出链长度 |

### 3.4 渐进式搬迁（动态过程）

扩容时不一次性迁移所有数据，而是**在每次 map 操作（增、删、改）时顺带搬迁 2 个桶**。

**搬迁流程**：
1. 创建新桶数组（`buckets`），旧桶数组存入 `oldbuckets`。
2. 每次访问时，检查当前正在访问的桶是否已搬迁，若未搬迁则搬迁该桶。
3. 同时后台或每次操作额外搬迁一个桶（具体数量版本不同）。
4. 所有桶搬迁完成后，`oldbuckets` 置为 nil。


**好处**：避免单次操作停顿时间过长（O(n) 搬迁分摊到 O(1) 操作）。

### 3.5 扩容期间的查询流程

当 `oldbuckets != nil` 时（扩容中），读取 key 的流程：

1. 计算哈希，定位桶索引。
2. 先检查 `oldbuckets` 中对应桶是否已搬迁：
   - **若未搬迁**：从旧桶中查找 key（因为数据还在旧桶）。
   - **若已搬迁**：直接从新桶中查找。
3. 返回结果。

写入和删除操作会先触发搬迁（确保数据在新桶），再执行操作。

### 3.6 扩容时的元素移动算法：高低位（翻倍扩容） vs 直接复制（等量扩容）
![高地位算法](https://img.draven.co/2020-10-18-16030322432585/hashmap-bucket-evacuate.png)

#### 翻倍扩容：使用**高低位算法**（rehash）

当 `B` 变为 `B+1` 时，桶数量翻倍。旧桶中的每个键值对需要决定迁移到新桶的哪个位置：

- 计算 `hash` 值。
- 旧桶索引 `oldIdx = hash & (2^B - 1)`。
- 新桶索引有两个可能：`oldIdx` 或 `oldIdx + 2^B`，由哈希值的**第 B 位**（即新增的那一位）决定：
  - 若该位为 0 → 迁移到 `oldIdx`
  - 若该位为 1 → 迁移到 `oldIdx + 2^B`

这个算法称为**高低位算法**（也叫增量哈希）。因为只需要检查哈希值的对应位，不需要重新计算完整模运算，效率极高。这也是为什么 Go map 扩容时能快速重分配的原因。

#### 等量扩容：不使用高低位算法，而是**直接复制+整理**

等量扩容时 `B` 不变，桶数量相同，因此不需要重新计算桶索引。但需要将旧桶中的数据紧凑地复制到新桶中：

- 创建新的 `buckets` 数组（大小与旧桶相同）。
- 遍历旧桶及其溢出桶，将每个键值对按照**原来的哈希值**重新插入新桶（实际上仍是低 B 位，结果同原桶索引）。
- 由于新桶没有溢出桶，每个桶的 8 个槽位填满后，会按需创建新的溢出桶（但会优先使用预分配的 `nextOverflow`）。
- 最终旧溢出桶全部释放，数据变得紧凑，查找效率恢复。

等量扩容不涉及高低位算法，本质是**内存碎片整理**。

## 4. 新老版本设计区别（Go 1.18 前后）

| 方面 | Go 1.17 及之前 | Go 1.18 及之后 |
|------|----------------|----------------|
| **溢出桶管理** | 按需 malloc，形成链表 | 预分配一批溢出桶（`mapextra` 结构优化），减少内存分配次数 |
| **扩容搬迁速度** | 每次操作搬迁 1 个桶 | 每次操作搬迁 2 个桶，加快扩容完成速度 |
| **遍历随机性** | 起始桶随机，但顺序相对固定（依赖插入顺序） | 增加伪随机因子，每次遍历顺序不同，防止用户依赖 |
| **哈希种子** | 每次创建 map 随机生成 | 增强随机性，防止哈希碰撞攻击 |
| **内存对齐** | 键值分开存储但部分版本对齐不佳 | 改进对齐，进一步减少内存浪费 |

## 5. 使用注意事项与常见陷阱

### 5.1 不要对 nil map 直接赋值

```go
var m map[string]int
m["key"] = 1  // panic: assignment to entry in nil map
```
**应对**：使用 `make(map[string]int)` 初始化。

### 5.2 非并发安全

多个 goroutine 同时读写 map 会触发 `fatal error: concurrent map writes`。
**应对**：使用 `sync.RWMutex` 或 `sync.Map`（适合读多写少场景）。

### 5.3 遍历顺序不确定，不可依赖

```go
m := map[int]string{1:"a", 2:"b", 3:"c"}
for k, v := range m {
    fmt.Println(k, v) // 每次运行顺序可能不同
}
```
**原因**：Go 有意设计随机起始桶，防止依赖顺序的 bug。

### 5.4 float 类型作为 key 的陷阱

浮点数精度问题可能导致看似相同的 key 无法匹配。
```go
m := map[float64]string{0.1: "ok"}
v, ok := m[0.1+0.2-0.2] // 可能等于 0.30000000000000004，不等于 0.1
```
**应对**：避免使用浮点数作 key，或使用固定精度格式化后作为字符串 key。

### 5.5 不能取 map 元素的地址

```go
v := m["key"]
addr := &v   // 可以（这是普通变量）
addr2 := &m["key"] // 编译错误：cannot take address of map element
```
**原因**：map 扩容时元素地址会变化，Go 禁止取址。

### 5.6 使用 `make` 预分配容量提升性能

已知元素数量时，预分配避免频繁扩容：
```go
m := make(map[string]int, 1000) // 提前分配足够桶
```

### 5.7 删除元素不会释放内存

`delete(m, "key")` 仅标记空位，底层桶不会自动缩容。若大量删除后内存占用高，可考虑新建 map。

### 5.8 值类型 map 的坑

若 map 的值是结构体，需要修改字段时不能直接赋值：
```go
type User struct{ Age int }
m := map[string]User{"a": {18}}
m["a"].Age = 20 // 编译错误：cannot assign to struct field
// 正确做法：
tmp := m["a"]
tmp.Age = 20
m["a"] = tmp
// 或使用 map[string]*User
```

### 5.9 比较 map 是否相等

map 不能直接使用 `==` 比较（除非与 `nil` 比较）。需要自定义循环比较。

## 6. 总结

Go 的 map 采用桶数组 + 溢出链结构，通过负载因子（6.5）和溢出桶数量触发翻倍或等量扩容，并采用渐进式搬迁避免停顿。翻倍扩容使用**高低位算法**快速重分配，等量扩容则直接复制整理。1.18 版本后优化了搬迁速度和溢出桶管理，增强了遍历随机性。使用时应特别注意并发安全、nil map、取址限制和浮点数 key 等问题。

> 注：本文档基于 Go 1.18~1.22 源码分析，具体实现细节可能随版本变化，建议查看目标版本的 `runtime/map.go`。
